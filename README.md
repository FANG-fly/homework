## 思考题1：波特率不一致实验

- **实验操作**：代码中 `Serial.begin(115200)`，串口监视器设置为 9600
- **观察现象**：串口监视器显示乱码（⸮⸮⸮...）
- **原因**：通信双方波特率不匹配，接收端无法正确解析数据位时序，导致解码错误
- **结论**：串口通信要求收发双方波特率必须一致

## 思考题2：修改 delay 数值

| delay 数值 | 周期 | 观察现象 |
|-----------|------|---------|
| 1000ms | 2秒 | LED 每秒闪烁 1 次（原始状态） |
| 200ms | 0.4秒 | LED 快速闪烁，每秒约 2.5 次 |
| 3000ms | 6秒 | LED 极慢闪烁，6秒完成一次亮灭 |

**原理**：`delay(n)` 使程序暂停 n 毫秒。LED 闪烁周期 = 高电平delay + 低电平delay，delay 越小，闪烁频率越高。

### lab04：触摸自锁开关

#### 实验内容

- 使用触摸引脚 (T0 / GPIO4) 作为输入

- 引入布尔型状态变量 (`bool ledState`) 记录 LED 状态
- 实现边缘检测逻辑（仅在"按下瞬间"触发一次状态翻转）
- 加入软件防抖（200ms 内忽略重复触发，防止手抖误判）

#### 关联基础

实验2 (基础IO) + 实验4 (触摸引脚)

#### 实验结果

- 开发板型号：ESP32 Dev Module
- 触摸引脚：T0 (GPIO4)
- LED 引脚：GPIO2 (板载 LED)
- 触摸阈值：40（根据实际开发板调整）
- 防抖延时：200ms
- 程序功能：摸一下触摸引脚，LED 亮起并保持长亮；松开手再摸一下，LED 熄灭

#### 核心逻辑

| 机制 | 实现方式 | 作用 |
| --- | --- | --- |
| 状态变量 | `bool ledState` | 记住 LED 是亮还是灭，实现"自锁"保持 |

| 软件防抖 | `currentTime - lastDebounceTime > 200ms` | 200ms 内的重复触发视为手抖，忽略 |

## ex07

Web网页端无极调光器（优化版）。

### lab07：Web 无极调光器

#### 实验内容

- 使用 `WiFi.h` 将 ESP32 配置为 SoftAP (无线热点) 模式
- 使用 `WebServer.h` 搭建 HTTP 服务器，基于课程规定的基础 HTML 模板托管网页
- 网页前端使用 HTML5 `<input type="range">` 滑动条控件
- 结合 JavaScript `fetch` API 实现 AJAX 无刷新通信，实时发送亮度参数
- 后端解析 HTTP GET 请求参数，通过新版 PWM API 动态调整 LED 亮度
- 响应头严格设置 `charset=UTF-8`，确保中文显示正常
- **前端优化**：添加 50ms 防抖 (Debounce) 机制与双事件监听，避免拖动时产生请求风暴
- **后端优化**：添加 10ms 最小请求间隔限流 (Throttling)，防止 ESP32 处理过载导致亮度回弹

#### 关联基础

实验3 (PWM呼吸灯) + Web 服务器基础 + 前端防抖与后端限流机制

#### 实验结果

- 开发板型号：ESP32 Dev Module
- LED 引脚：GPIO2 (板载 LED)
- WiFi 模式：SoftAP (热点名称: ESP32_Dimme, 密码: 12345678)
- 默认 IP 地址：192.168.4.1
- 程序功能：手机或电脑连接 ESP32 热点后，在浏览器中拖动网页滑动条，可无刷新、无延迟、无回弹地实时控制 LED 亮度 (0-255)，并带有操作状态反馈

#### 核心逻辑

| 机制 | 实现方式 | 作用 |
| --- | --- | --- |
| AP 热点模式 | `WiFi.softAP()` | ESP32 独立发射 WiFi，摆脱外部路由器限制，便于演示 |
| AJAX 无刷新 | JS `fetch()` API | 后台静默发送 HTTP 请求，避免网页重载导致的卡顿 |
| 字符编码规范 | `charset=UTF-8` | 在 `server.send` 响应头中指定，防止网页中文乱码 |
| 客户端防抖 | JS `setTimeout` / `clearTimeout` | 将高频滑动事件合并，减少 80% 的无效网络请求 |
| 服务器限流 | `millis()` 时间戳比对 | 强制最小请求间隔，防止 ESP32 处理队列堆积导致亮度回弹 |

## ex08

物联网安防报警器模拟实验（触摸传感器优化版）。

### lab08：物联网安防报警器

#### 实验内容

- 使用 `WiFi.h` 开启 SoftAP 热点模式，构建独立的物联网安防局域网
- **结合 ESP32 内部电容式触摸传感器 API (`touchRead`)**，使用 GPIO4 (T0) 模拟门窗磁/人体入侵传感器
- Web 端提供完整的安防控制台 UI（包含状态指示、布防/撤防控制）
- **前端核心技术**：使用 JavaScript `setInterval` 结合 `fetch` 实现 **AJAX 定时轮询 (Polling)**，网页每秒自动获取 ESP32 状态，实现无刷新实时报警弹窗
- **后端核心技术**：引入系统状态机逻辑（撤防、布防、报警），在 `setup()` 中加入触摸环境基准值校准机制，在 `loop()` 中独立处理硬件传感器中断与 Web 网络请求
- **交互优化**：串口监视器自动打印完整的 `http://` 网页访问地址，方便快速测试；触摸触发阈值默认设定为 500，适配不同环境

#### 关联基础

实验2 (GPIO输入/输出) + ESP32 触摸传感器特性 + 实验7 (WebServer与AJAX) + 状态机编程思想

#### 实验结果

- 开发板型号：ESP32 Dev Module
- 传感器：GPIO4 / T0 (电容式触摸引脚，手指触摸模拟入侵触发，阈值 < 500)
- 执行器：GPIO2 (板载 LED，模拟报警闪烁灯)
- WiFi 模式：SoftAP (热点名称: ESP32_Security, 密码: 12345678)
- 默认 IP 地址：192.168.4.1
- 程序功能：串口自动输出完整网页URL；网页可远程布防/撤防；布防后一旦手指触摸 GPIO4 引脚，ESP32 LED 狂闪，网页无需手动刷新即可在 1 秒内自动变红并显示报警信息，支持远程一键解除报警。

#### 核心逻辑

| 机制 | 实现方式 | 作用 |
| --- | --- | --- |
| 状态机模型 | `systemState` 变量 (0/1/2) | 清晰界定撤防、布防、报警三种状态，防止逻辑冲突 |
| 触摸传感检测 | `touchRead(TOUCH_PIN) < 500` | 读取电容值变化，手指触摸时数值骤降，低于阈值即触发报警 |
| 环境自适应校准 | `setup()` 中读取基准值 | 开机时获取环境基准值并打印，方便开发者微调触发阈值 |
| 串口快捷引导 | 拼接打印 `http://` + IP | 开发者可直接复制串口输出的网址到浏览器，提升调试效率 |
| AJAX 轮询 | JS `setInterval` (1000ms) | 模拟物联网设备的实时心跳监控，网页自动刷新状态 |
| 异步报警闪烁 | `loop()` 中的电平翻转 + `delay` | 在报警状态下独立控制 LED 闪烁，不阻塞 Web 服务器的请求处理 |
